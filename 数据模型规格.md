# 签证 CRM —— 数据模型规格（给 Claude Code 重写 schema 用）

> **给 Claude Code 的指令**:之前的 schema 把需求过度简化了。请严格按本规格重写 `supabase/migrations/0001_init.sql` 和 `src/types/domain.ts`,**不要保留旧的通用枚举(tourist/business/work 等),不要自由发挥**。保持原有的优点:幂等(可重复执行)、`set_updated_at` / `handle_new_user` / `is_admin` / `prevent_profile_escalation` 这几个函数与触发器照旧保留。所有金额用 `numeric(12,2)`,所有主键用 `uuid default gen_random_uuid()`(profiles 除外,它引用 auth.users)。

---

## 0. 保留不动的部分

以下来自当前 schema,**保持不变**:

- `app_role` 枚举(admin / staff)
- `profiles` 表 + 它的三个触发器(updated_at、防提权、新用户自动建档)
- `set_updated_at()`、`handle_new_user()`、`is_admin()`、`prevent_profile_escalation()` 函数
- 回填现有 auth.users 为 staff 的逻辑
- 不硬编码任何 admin 邮箱

## 1. 枚举类型（全部重定义）

| 枚举名 | 值 | 说明 |
|---|---|---|
| `app_role` | `admin`, `staff` | 保留 |
| `case_stage` | `consulting`, `preparing`, `nomination_lodged`, `nomination_approved`, `visa_lodged`, `additional_docs`, `granted`, `refused`, `withdrawn` | 对应:咨询→准备材料→递提名→提名批→递签证→补件→授签/拒签/撤案。189/190 无提名,直接 preparing→visa_lodged |
| `lodgement_type` | `nomination`, `visa` | 提名 / 签证 |
| `lodgement_outcome` | `pending`, `approved`, `refused`, `withdrawn` | 单条递交的结果 |
| `payment_direction` | `from_client`, `to_company` | 客户付给你 / 你付给主代理（双流账目核心） |
| `payment_method` | `cash`, `transfer`, `wechat`, `alipay`, `card`, `other` | 保留 |
| `follow_up_channel` | `call`, `wechat`, `email`, `meeting`, `other` | 保留 |
| `doc_type` | `passport`, `medical`, `police_check`, `english_test`, `photo`, `employment`, `financial`, `form`, `other` | 含体检/无犯罪/英语成绩 |

> `case_category`（案件大类,2026-06 增,迁移 0032）同理**不做 DB 枚举**,可空 `text`。取值由前端常量 `CASE_CATEGORIES`(types/domain.ts) 约束:签证申请 / 职业评估 / De Facto 关系认定 / 定制文件。与 `visa_subclass`(UI 统称「案件类型」)构成「大类(粗)→类型(细)」两级,相互独立、不级联;旧案件留空。

## 3. Dashboard 查询逻辑（验证 schema 够用）

这几个查询必须都能算出来,作为 schema 完整性的验收标准:

- **处理时间快到的案件**:`lodgements` 中 `outcome='pending'` 且 `(lodged_date + dha_processing_days) - current_date <= N`(N 可配,如 14)
- **文件快过期**:`documents` 中 `is_archived=false` 且 `expiry_date is not null` 且 `expiry_date - current_date <= 30`
- **逾期未付款**:`installments` 中 `is_paid=false` 且 `due_date < current_date`
- **优先客户**:`customers` 中 `is_starred=true`,再按 `priority_tier`(vip→a→b→c,NULL 最后)排序
- **客户欠款**:`payment_plans.client_total - sum(payments where direction='from_client')`
- **欠主代理**:`payment_plans.company_total - sum(payments where direction='to_company')`

## 4. RLS 策略

延续"团队全信任"模型,但**用软删除**:

- **业务表**(employers / customers / cases / lodgements / case_stage_history / documents / payment_plans / installments / payments / follow_ups / tasks):
  - `SELECT` / `INSERT` / `UPDATE` → `to authenticated using (true) with check (true)`
  - **`DELETE` → 仅 admin**(`using (public.is_admin())`)。日常"删除"= 把 `is_archived` 设为 true,不真删
- **profiles**:保持现状(全员可读、admin 可全写、本人可改自己 + 防提权触发器)

所有业务表 `enable row level security`。

---

## 5. Storage

- 私有 bucket `case-files`(public=false),幂等创建
- 策略:`to authenticated`,`bucket_id='case-files'`,未登录→403
- 文件路径建议按 `case_id/文件名` 组织;前端用 `createSignedUrl` 给短时链接,不暴露公开 URL

---

## 6. Realtime

把这些表纳入 `supabase_realtime` 发布(用于实时刷新 Dashboard 与看板):`cases`、`lodgements`、`tasks`、`installments`。用存在性判断包裹,避免重复添加报错。

## 8. 待你朋友确认的点（不确定就先用括号里的默认值）

1. **货币**:默认 `AUD`。她的客户主要付 AUD 还是 CNY?会不会同一个案子既收 CNY 又付主代理 AUD(涉及汇率)?**v1 建议:每个案子/计划锁定一种货币,不做自动汇率换算。** 如果必须多币种,告诉我再加汇率字段。
2. **客户等级**:默认 `vip/a/b/c`。她想要几级?各级只是排序优先,还是有功能差异?
3. **附属申请人范围**:配偶 + 子女?还是父母、成年子女也算?(影响 relationship_to_primary 的取值)
4. **任务负责人**:1-2 人的话任务要不要分配给特定人,还是大家共享一个待办池?
5. **拒签复议(AAT/ART)**:要不要追踪?要的话给 case_stage 加 `appeal` 阶段或单独建表。
6. **审计日志**:本规格暂未含全局审计日志(只有 case_stage_history)。要不要现在就加一张 `audit_log`?小团队 v1 可先不加。
