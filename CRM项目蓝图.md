# 移民签证 CRM — 项目蓝图

> 给朋友搭建的澳洲移民代理案件管理系统。非商用、内部使用、1–2 人、全平台浏览器可用。
>
> 本文档是项目的唯一事实来源（single source of truth）。建议存进仓库 `docs/blueprint.md`，每个开发阶段回头核对，也可直接作为规格书提供给 Claude Code。

---

## 1. 项目概述

这是一个**垂直领域**的 CRM，专门服务澳洲移民/签证代理业务。它不是通用销售 CRM，而是一个**案件管理系统（case management）**，管理客户签证申请的全生命周期。

**和通用 CRM 的本质区别：**

- 核心实体是「客户 + 案件 + 递交记录」，而不是「线索 + 商机」
- 重度依赖**时间预警**：DHA 处理时间、文件到期、付款逾期
- 需要建模**主副申请人的家庭关系**
- 付款是**双流账目**（客户付给代理 / 代理付给主代理）

**目标用户：** 朋友（移民代理）+ 可能的少量员工
**使用场景：** 电脑端日常办公 + 手机端随时查看与更新

## 2. 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 前端框架 | Vite + React + TypeScript | 纯 SPA，保留现有脚手架 |

| 部署 | Vercel（静态托管） | 自动 HTTPS + 全球 CDN |

## 3. 系统架构

前端（Vite + React）运行在用户浏览器中，由 Vercel 静态托管。所有数据操作通过 Supabase JS SDK 直连 Supabase 平台。

```
┌─────────────────────────────────────┐
│         前端 (Vite + React)          │
│       浏览器运行，Vercel 托管         │
└───────────────┬─────────────────────┘
                │ API 调用 (HTTPS)
                ▼
┌─────────────────────────────────────┐
│           Supabase 平台              │
│  ┌────────┐ ┌────────┐ ┌────────┐  │
│  │  Auth  │ │Database│ │Storage │  │
│  └────────┘ └────────┘ └────────┘  │
│  ┌──────────────┐ ┌────────────┐   │
│  │Edge Functions│ │  Realtime  │   │
│  └──────────────┘ └────────────┘   │
└─────────────────────────────────────┘
```

各服务在本 CRM 的职责：

- **Auth** — 朋友和员工的登录、密码、会话保持
- **Database** — 客户档案、案件、付款、跟进记录全部存这里
- **Storage** — 护照扫描件、体检报告、合同 PDF
- **Edge Functions** — 管理员建账号、定时抓 DHA 处理时间、扫描快过期文件
- **Realtime** — 多设备同时打开时数据自动同步

**核心安全原则：RLS-first** — 所有数据访问权限都在数据库层用 Row Level Security（行级安全）规则定义。前端假设「凡是查得到的就是允许看的」，不在前端做安全判断。`service_role` key 只出现在 Edge Functions 中，永远不进入前端代码或代码仓库。

## 4. 数据模型

### 核心实体

**Client（客户）**
- 基本信息：姓名、生日、护照号、护照到期日、联系方式
- `sponsor_employer_id` → 关联担保雇主
- `primary_applicant_id` → 自引用：主申请人此字段为空，副申请人指向主申请人
- `priority_tier` → 客户等级（VIP / A / B / C）
- `is_starred` → 是否标注优先

**Employer（担保雇主）**
- 独立表，一个雇主可担保多个客户
- 雇主名称、ABN、联系信息、担保状态

**Case（签证案件）**
- 一个客户可有多个案件
- `visa_subclass` → 签证类型（482 / 186 / 189 / 190 …）
- `current_stage` → 当前阶段枚举（准备材料 → 递提名 → 提名批 → 递签证 → 补件 → 授签/拒签）
- 关联 Client，雇担类签证关联 Employer

**Lodgement（递交记录）** — 关键设计
- 一个案件下挂 1–2 条（482/186 有提名+签证两条；189/190 仅签证一条）
- `type` → nomination / visa
- 递交日期、参考号、DHA 处理时间（手动录入，天数）
- 派生计算：已递天数、剩余天数、进度条颜色（绿/黄/红 = 剩余 >50% / 20–50% / <20%）

**Document（文件）** — 分两层
- Client 级：护照、出生证明等跟人走的
- Case 级：体检、无犯罪、英语成绩等为某个具体申请做的
- 字段：类型、签发日期、到期日期、预警阈值（默认 30 天）、文件 URL

**PaymentPlan + Installment（付款计划与分期）**
- 每个案件一个 PaymentPlan
- **双流账目**：
  - `client_payable` / `client_paid` —— 客户欠你 / 已付你
  - `company_payable` / `company_paid` —— 你欠主代理 / 已付主代理
- Installment（分期节点）：到期日、金额、状态（待付/已付/逾期）

**Activity（跟进记录）**
- 关联 Client（必填）+ Case（可选）
- 类型：电话 / 微信 / 邮件 / 会议 / 备注
- 内容、时间、操作人

**Task（任务）**
- 关联 Client / Case
- 标题、到期日、状态、负责人

**AuditLog（审计日志）**
- 记录谁、何时、改了什么
- **第一天就建**，事后补极其痛苦

### 实体关系

- Client `1—N` Case
- Client `N—1` Employer
- Client `1—N` Client（主 → 副申请人，家庭组）
- Case `1—N` Lodgement
- Case `1—1` PaymentPlan `1—N` Installment
- Client / Case `1—N` Document
- Client / Case `1—N` Activity
- Client / Case `1—N` Task

## 7. 账号与所有权

**原则：所有账号归朋友所有，你以协作者身份参与，交付后可干净退出。**

- 朋友用她的 Gmail（Google 登录）注册 Supabase 和 Vercel

## 8. 安全要求

平台（Supabase / Vercel）本身是企业级安全（SOC 2、AWS、默认加密、HTTPS）。真正的风险在使用方式，需逐条落实：

- [ ] 所有敏感表配置严格 RLS policy（只有授权用户能看对应客户）
- [ ] Storage bucket 全部 private，文件用 signed URL 临时授权（链接数小时过期）
- [ ] `service_role` key 只在 Edge Functions 使用；`.gitignore` 包含 `.env*`，绝不 commit
- [ ] Gmail / Supabase / Vercel / GitHub 全部开启 2FA（开发者和朋友都要开）
- [ ] 生产域名启用 HTTPS（Vercel 自动）
- [ ] 开启 Supabase 自动备份（Free tier 默认 7 天 PITR）
- [ ] 每月手动 export 一份数据到朋友自己的硬盘做长期备份
- [ ] 凭证（URL / key）通过私密渠道传递，不发群、不截图外传

**法律提示：** 处理澳洲客户的护照与签证信息受《Privacy Act 1988》约束，即使非商用也有数据知情、保护、不转让、泄漏响应等义务。若朋友将来正式经营，应咨询 MARA（移民代理注册局）与 OAIC（隐私专员办公室）的合规要求。

## 9. 待向朋友确认的需求

这些细节会影响数据库 schema，**建议在 Phase 2 开始前确认**：

1. **客户分等级**：几级？等级之间只是排序差异，还是有功能区别（如 VIP 专属预警）？
2. **标注客户**：只有 admin 能标，还是每个员工有自己的标注？
3. **货币与汇率**：金额都是 AUD 吗？是否有客户付人民币、需要汇率换算？
4. **附属申请人范围**：仅配偶+未成年子女，还是父母、成年子女也算？
5. **拒签复议（AAT/ART）**：要不要在系统里追踪？独立流程还是案件加状态？
6. **收据/发票**：每笔付款要不要 CRM 自动生成 PDF？还是用 Xero/MYOB 等外部工具？
7. **文件大小限制**：常见扫描件几 MB？是否需支持 100MB+ 大文件（影响存储策略）？
8. **多语言**：界面只做中文，还是中英双语切换？
9. **数据导出**：是否需要把客户清单、付款记录导出成 Excel（报税、对账常用）？
